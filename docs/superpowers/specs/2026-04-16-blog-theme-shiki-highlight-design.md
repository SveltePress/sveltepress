# Blog Theme Shiki Code Highlighting

Full code block highlighting for `@sveltepress/theme-blog`, matching `@sveltepress/theme-default` feature set.

## Context

The blog theme (`packages/theme-blog/`) has basic Shiki syntax highlighting via a `rehypeShiki` plugin, but lacks all the rich code block features present in the default theme: code commands, title bars, line numbers, copy buttons, and Twoslash integration.

Blog posts are pre-rendered to HTML strings at build time and stored in virtual modules. They are rendered in `PostLayout.svelte` via `{@html post.contentHtml}`, which means Svelte components embedded in the HTML will NOT be compiled. This is the key architectural constraint.

## Architecture: Shared Code in `@sveltepress/vite`

Highlighting logic is shared between themes — NOT duplicated. Shared code lives in `@sveltepress/vite` under a new `src/highlight/` directory, exported via `@sveltepress/vite/highlight`.

Rationale for `@sveltepress/vite` over a new package:
- Both themes already depend on it — zero new dependency edges
- `shiki`, `lru-cache`, `unified`, `unist-util-visit` are already its dependencies
- The `Highlighter` type is already defined here
- Code highlighting is a build concern, and vite is the build core
- Avoids a new package to version, publish, and maintain

### What is shared (in `@sveltepress/vite`)

1. **`commands.ts`** — command parsing and overlay div generation (moved from `theme-default/src/markdown/commands.ts`)
2. **`code-block.ts`** — `prepareCodeBlock()` and `wrapCodeBlock()` utilities

### What stays in each theme

- **theme-default**: Shiki init with Twoslash Svelte renderer, `<CopyCode />` Svelte component, UnoCSS styles, `{`/`}` escaping for Svelte compilation
- **theme-blog**: Shiki init with optional plain Twoslash, HTML `<button>` + event delegation for copy, plain CSS, NO curly brace escaping

### Shared API

```typescript
// @sveltepress/vite/highlight

// --- commands.ts (moved from theme-default) ---
export { processCommands, COMMAND_RE, COMMAND_CHEAT_LIST }

// --- code-block.ts (new) ---
export interface PreparedCodeBlock {
  /** Code with commands stripped, ready for Shiki */
  processedCode: string
  /** Command overlay HTML divs */
  commandDoms: string[]
  /** Title from meta, if present */
  title?: string
  /** Whether line numbers were requested (ln in meta) */
  containLineNumbers: boolean
  /** Original lines array (for line number generation) */
  lines: string[]
  /** Whether @noErrors was present as first line */
  noErrors: boolean
}

export interface WrapCodeBlockOptions {
  /** HTML for the copy button. Each theme provides its own. */
  copyButtonHtml?: string
  /** Escape { } to &#123; &#125; for Svelte compilation. Default: false */
  escapeForSvelte?: boolean
}

/**
 * Parse code block metadata and process code commands.
 * Call BEFORE Shiki highlighting.
 */
export function prepareCodeBlock(code: string, meta?: string): PreparedCodeBlock

/**
 * Wrap Shiki-highlighted HTML in the standard code block structure.
 * Call AFTER Shiki highlighting.
 */
export function wrapCodeBlock(
  highlightedHtml: string,
  lang: string,
  prepared: PreparedCodeBlock,
  options?: WrapCodeBlockOptions
): string
```

## Pipeline Change (Blog Theme)

```
Before: remarkParse -> remarkGfm -> remarkRehype -> rehypeShiki -> rehypeStringify
After:  remarkParse -> remarkGfm -> remarkCodeBlocks (new) -> remarkRehype -> rehypeStringify
```

The new `remarkCodeBlocks` remark plugin intercepts `code` nodes (which carry `lang`, `meta`, `value`), processes them via shared utilities, and replaces them with `html` nodes containing the full code block HTML.

Both `remarkRehype` and `rehypeStringify` must be configured with `allowDangerousHtml: true` so that raw HTML nodes from the remark plugin pass through to the final output. This is safe because the HTML is generated by us, not user-supplied.

## Features

### 1. Code Commands

Uses `processCommands()` from `@sveltepress/vite/highlight`. Same syntax, same output as default theme.

| Command | Purpose | Example |
|---------|---------|---------|
| `hl` / `~~` | Highlight lines | `// [svp! hl:3]` |
| `++` | Diff add | `// [svp! ++]` |
| `--` | Diff remove | `// [svp! --]` |
| `df` | Diff generic | `// [svp! df]` |
| `fc` / `!!` | Focus (blur rest) | `// [svp! fc:2]` |

### 2. Title Bar

Handled by `prepareCodeBlock()` — extracts from meta: `` ```ts title="config.ts" ``

### 3. Line Numbers

Handled by `prepareCodeBlock()` — detects `ln` in meta: `` ```ts ln ``

### 4. Copy Button

Blog theme provides an HTML button (SVG icons) to `wrapCodeBlock()` via `copyButtonHtml`. Event delegation in `PostLayout.svelte`'s `onMount` handles click behavior.

### 5. Twoslash

Uses `@shikijs/twoslash` with a plain HTML renderer (`rendererRich`) instead of the default theme's Svelte-specific renderer. Produces pure HTML/CSS hover tooltips that work inside `{@html}`. Optional, configured via `highlighter.twoslash: true`.

### 6. LRU Cache

200-entry LRU cache for development mode, same as default theme.

## HTML Output Structure

Identical to default theme (produced by shared `wrapCodeBlock()`):

```html
<div class="svp-code-block-wrapper">
  <div class="svp-code-block--title">file.ts</div>
  <div class="svp-code-block svp-code-block--with-line-numbers">
    <div class="svp-code-block--command-line svp-code-block--hl"
         style="top: calc(2 * 1.5em + 12px);"></div>
    <pre class="shiki shiki-themes vitesse-light night-owl">
      <code>...</code>
    </pre>
    <div class="svp-code-block--lang">ts</div>
    <!-- copy button: theme-specific HTML -->
    <div class="svp-code-block--line-numbers">
      <div class="svp-code-block--line-number-item">1</div>
    </div>
  </div>
</div>
```

## CSS Strategy

All styles in `GlobalLayout.svelte` using `:global()` selectors, adapted to blog theme CSS variables (`--sp-blog-*`). Existing Shiki dual-theme CSS is retained.

## Testing Plan

- **Unit tests** in `@sveltepress/vite`: commands, prepareCodeBlock, wrapCodeBlock
- **Integration tests** in `theme-blog`: full remark pipeline producing expected HTML
- **Regression tests**: existing `theme-default` tests must pass after refactor
- **Manual verification**: build `example-blog` and check rendering in browser

## Scope Boundary

This design covers:
- Extracting shared highlighting code to `@sveltepress/vite`
- Refactoring `theme-default` to consume shared code (minimal, import-path changes only)
- Implementing full code highlighting in `theme-blog`

It does NOT change:
- Any theme-default behavior or output
- Blog theme post parsing, virtual modules, RSS, scaffolding, or theme toggle
